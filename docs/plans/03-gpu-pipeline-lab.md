# Phase 3 — GPU pipeline, pipeline lab, fixture tooling, pilot recording

## Goal

The full GPU reduction stage per `detection.md` runs on-device, visualized live in a hidden `/lab` route; replay is deterministic; the fixture system (canonical raw ROI clips + annotation sidecars, regenerable strip-energy JSON) exists with field-proven capture and export; and a short race-speed pilot recording has validated that the strip signal resolves a real whoop crossing at all — the ADR 0003 go/no-go — before the full corpus is gathered.

## Scope

**In:** `FrameSource` abstraction on WebCodecs decode, canonical-texture import design, complete WGSL pipeline (dt-scaled EMA, integer strip counts), device-loss recreation, synthetic frame generator, golden tests, fixture formats + recorder + regeneration tool + annotation tooling, frame ring buffer, `/lab` debug UI with touch ROI + wake lock + on-device self-test, UI bridge pattern, orientation policy, pilot recording checkpoint, rolling corpus kickoff.

**Out:** the crossing state machine (Phase 4) — `/lab` shows raw strip energies only. Product screens. Persistence (fixtures leave the device via share/download, not OPFS writes — keeping the storage seam clean). The *completed* 30+ event corpus is explicitly **not** a Phase 3 exit criterion (see item 15).

## Work items (dependency order)

1. **`FrameSource` abstraction FIRST** — the pipeline consumes an interface, never `getUserMedia` directly:
   - `CameraSource` (Phase 2's CameraService + rVFC).
   - `VideoFileSource` **built on WebCodecs `VideoDecoder`** (available on both target platforms) for frame-exact, timestamp-exact stepping. HTMLVideoElement playback is non-deterministic (drops, compositor timing) and is allowed only as a live-preview convenience, never for tests, annotation, or CI. Replay **honors recorded inter-frame capture-timestamp gaps** so a live EMA trajectory (including dropped frames) is reproduced exactly.
   - `SyntheticSource` (programmable moving blob over static/noisy backgrounds with known ground-truth crossing frames).
2. **Canonical-texture import design** (resolves the shader-variant drift risk): a minimal per-source **import/normalization pass** converts any input — external texture, copied texture, decoded `VideoFrame`, synthetic texture — into one canonical downscaled luminance `texture_2d`. Everything downstream (EMA, threshold, strip reduction) is a **single shared WGSL path** fully covered by golden tests; only the thin import pass differs per platform/source, verified by on-device smoke checks (and, per the Phase 2 constructed-`VideoFrame` result, run in CI where possible). If iOS forces the copy fallback, amend `detection.md`'s "frames enter WebGPU as external textures" line now.
3. **Versioned fixture formats, frozen now — with clip primacy declared:**
   - **Canonical corpus artifact: raw downscaled ROI clips + JSON annotation sidecars.** Clips are stored **lossless or near-lossless** (raw frames or a lossless codec — small at ~256 px ROI): lossy quantization noise passes through the threshold nonlinearity and diverges replay from live near the trigger level. Decide the codec plus a **quantified live-vs-replay energy divergence bound** before any corpus recording. Sidecars carry ground truth (crossing frame indices + directions), recording conditions, and a **tier**: `must-pass` vs `known-limitation` — the tier split defines Phase 4's acceptance gate before tuning starts, and lets hard field fixtures be committed without breaking CI.
   - **Strip-energy JSON is a regenerable derivative, not ground truth:** `{ formatVersion, tunables (provenance), frames: [{ captureTimeMs, energies[] }] }`. Energies embed the tunables and pause behavior active at generation time; annotations attach to clips/frame indices, never to energy JSON.
   - **Repo storage decision, made with the format freeze:** binary clips in plain git grow the clone forever — pick git LFS or a size budget/downscaled-only policy.
4. **`FrameSample` contract** — `{ captureTimeMs, energies: Float32Array }` — the typed GPU→CPU seam every later layer builds against.
5. **WGSL pipeline** on the canonical texture: EMA background, abs-diff threshold to binary mask, per-strip reduction. Two decisions written into `detection.md`:
   - **EMA alpha is defined per unit time** (dt-scaled from inter-frame capture timestamps), not per frame — otherwise the effective time constant differs between 60 fps and 30 fps devices, and dropped frames diverge live from replay. The spec's "~0.05" default is restated as a time constant.
   - **Strip energy is an integer hot-pixel count** (`u32` atomicAdd over the binary mask — integer addition is order-independent), normalized by strip pixel count on the CPU. Determinism is exact, normalization handles uneven strip widths, and golden-test tolerances become equalities.
6. **GPU device-loss handling:** `device.lost` listener + full pipeline re-creation path, informed by Phase 2's observations; testable by forcibly destroying the device.
7. **GPU golden tests in CI** (the Phase 1 rig): hand-constructed input textures with known blob positions → expected strip counts (exact); determinism test (same frames → identical energies); `SyntheticSource` sequences asserting the expected wave shape, that a paused EMA does not absorb a stationary blob, and that **a sequence with dropped frames produces energies matching the gap-free equivalent within tolerance** (validating dt-scaling).
8. **Frame ring buffer** (last ~2 s of downscaled ROI frames) — the CLAUDE.md video-capture seam, built now and *exercised* as the clip recorder below. Test asserts it holds the last K frames.
9. **Fixture recorder behind a debug flag** (ships in this first camera-capable build and stays in production builds): raw ROI clips via the ring buffer **plus a continuous-clip mode** for long events (lighting changes, walk-behinds, EMA re-adaptation outlast 2 s), and strip-energy JSON as a convenience capture. Anchor-download suffices on Android/desktop; if recording on iOS, use the Web Share API with files (anchor-download is unreliable there — best-effort, and Phase 7's export wants the same API anyway).
10. **Fixture-regeneration tool:** clip → `VideoFileSource` → GPU → strip-energy JSON, runnable locally and in the GPU CI leg. Whenever tunables or the (future) pause contract change, JSON caches are re-derived from the canonical clips. This is what makes "clips canonical, JSON derived" operational rather than aspirational.
11. **Annotation tooling:** a frame-stepper mode in `/lab` — load a clip, step frame by frame (WebCodecs makes this exact), mark crossing frames + directions + tier, emit the sidecar. Hand-editing JSON against video at ±1-frame precision would corrupt the acceptance dataset.
12. **`/lab` route** (thin Svelte over the framework-free pipeline): live preview, touch-draggable/resizable ROI (prototyping the product calibration interaction early), live strip-energy bars with adjustable trigger line, EMA/threshold/strip-count controls, fps/latency readout, record buttons, **wake lock held whenever a camera source is active** (reacquired on `visibilitychange` — the Phase 3/4 field sessions are minutes of hands-off capture), and the **on-device GPU self-test panel**: run the embedded golden vectors and one bundled corpus clip through the device's own GPU, report pass/fail and max deviation vs committed expectations. This is the only possible automated GPU check on real iOS Metal, where WGSL numeric behavior can silently diverge from Dawn/Vulkan.
13. **UI bridge pattern, established here as THE rule:** per-frame data (strip energies, preview) is drawn **directly to canvas from the frame loop**, bypassing Svelte reactivity; `$state`/`.svelte.ts` bridges are reserved for low-frequency state (tunables, status). Pushing 30–60 Hz updates through the reactivity graph is a jank/battery trap that would get copied into the armed screen.
14. **Backpressure + orientation policies** (written into `detection.md`):
    - If readbacks fall behind the camera, drop frames at the source rather than queueing; downstream logic tolerates gaps (all windows in milliseconds, never frame counts).
    - Orientation: **an app-state binding, not an OS screen lock** (`screen.orientation.lock()` is unavailable on iOS Safari and needs fullscreen/installed context elsewhere). The session's ROI is bound to the orientation captured at setup; on orientation change, warn and invalidate detection until restored. A real screen lock is at most a Phase 7 installed-PWA enhancement.
15. **Pilot recording checkpoint → ADR 0003 go/no-go:** once the pipeline and `/lab` are minimally live, record a handful of real race-speed fly-throughs and review signal quality offline — strips-per-frame advancement (at 3–6 m/s a crossing spans only ~3–6 frames at 60 fps; the "wave" may advance 2–4 strips per frame or skip strips, heavily motion-blurred), SNR above the noise floor, blur behavior. A hand-wave is orders of magnitude larger and slower and proves nothing about this. Output: camera-placement/strip-count guidance for corpus recording, or an ADR 0003 revision while zero downstream investment exists.
16. **Rolling corpus recording (started here, NOT a Phase 3 exit criterion):** using the pilot-validated setup, accumulate 30+ annotated events across conditions — race-speed fly-throughs both directions, slow/angled passes, hand-waves, pilot walking behind, lighting changes, **AE/AF-transient scenarios per the Phase 2 lockability evidence** (drone close to lens, bright clothing entering frame, focus hunts), LED-lit vs dark whoop. This needs drone, pilot, venue — a calendar item that must not stall engineering. Completion gates **Phase 4's acceptance/tuning items only** (its entry criterion), not Phase 4's start.

## Verification (exit criteria — all drone-free except the pilot checkpoint)

- CI: GPU golden (exact), determinism, dropped-frame-EMA, and regeneration-tool tests green in headless Chromium WebGPU.
- **Deterministic replay proven:** the same clip through `VideoFileSource` yields a bit-identical `FrameSample` sequence across runs.
- **Replay fidelity proven on hand-wave fixtures:** a recorded clip replayed through the full pipeline produces strip energies matching the live-recorded JSON within the declared divergence bound.
- On-device on the Android phone: `/lab` bars track a hand-wave at camera rate; the on-device self-test passes; wake lock holds through a 5-minute hands-off capture; recorder export works; forced device loss recovers. Same checks on iOS if a device is available (best-effort, non-gating; share-sheet export is the item most likely to need iOS-specific work).
- Pilot recording reviewed; ADR 0003 go/no-go decision recorded.

## Risks retired

- **Detection feasibility on a real whoop** — the pilot checkpoint answers whether the strip signal temporally resolves a race-speed crossing *before* the scarce corpus is recorded with a possibly-inadequate setup and before any state-machine investment.
- **Shader-variant drift** — one golden-tested WGSL path serves every source and platform; only a thin import pass varies.
- **Un-replayable / non-deterministic pipeline** — WebCodecs decode + lossless clips + dt-scaled EMA make all future tuning drone-free, deterministic, and faithful to live behavior.
- **Corpus invalidation by tuning** — clips are canonical and tunable-independent; JSON is regenerated, never trusted stale.
- **Silent Metal/Vulkan numeric divergence** — the on-device self-test checks real-device GPU output against committed expectations.
- **Fixture scarcity / harvest-loop death** — recorder + tiers exist before they are needed; hard cases can be committed without breaking CI.
- **Ring-buffer seam rot, touch-ROI UX risk, per-frame reactivity jank, mid-capture screen lock** — all exercised or ruled here, not discovered later.
