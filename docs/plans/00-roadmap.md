# ChronoWhoop build roadmap (v2 — post expert review)

Seven phases, riskiest-first. Every phase ends with something demoable on a real phone at a deployed HTTPS URL, or verifiable in CI. Source of truth for behavior remains `docs/specs/`; accepted ADRs (phone-first PWA, WebGPU-only, strip differencing, OPFS behind an interface, Svelte 5 + Vite + TS) are not revisited here.

> **ADR 0006 (post-roundtable):** iOS is best-effort, not a gating platform. Android Chrome + desktop Chromium gate every phase; wherever this roadmap or a phase doc says "both phones"/"both platforms", read "Android required; iOS additionally if a device is available". iOS failures produce documentation, not pivots.

> **ADR 0009 (post-Phase-2 spike):** the required target device (S22/Xclipse) has no stock WebGPU; the reduction stage is CPU over WebCodecs capture, superseding ADR 0002. Wherever Phases 3–7 say WGSL/GPU/readback/device-loss/"GPU CI leg", read the CPU-pipeline equivalent per each phase doc's ADR 0009 amendment; the GPU-golden CI rig is retired in favor of plain node determinism tests. Phase 3's doc is rewritten; 4–7 carry amendment preambles.

## Phase list

| # | Phase | One-line goal |
|---|---|---|
| 1 | [Foundation](01-foundation.md) | Deployed HTTPS shell with capability gate, enforced module seams, visible build id, and a CI that provably runs every test category the project will ever need |
| 2 | [Device-risk spike](02-device-spike.md) | A permanent `/diag` screen proving camera → WebGPU → readback, capture timestamps, speech patterns, atomic-write behavior, and OPFS behavior on real Android Chrome hardware against pre-declared go/no-go thresholds — iOS measured informationally if a device is available |
| 3 | [Detection pipeline + pipeline lab + fixture tooling](03-gpu-pipeline-lab.md) | The full CPU reduction stage (single shared path behind the `FrameSource` abstraction, per ADR 0009) live in a debug lab, deterministic bit-exact clip replay, the canonical clip+annotation fixture system, and a race-speed pilot recording validating ADR 0003 |
| 4 | [Crossing detection](04-crossing-detection.md) | Precisely specified, fixture-validated crossing state machine; standing at the gate, test mode beeps on every correct-direction crossing and never on noise |

## Sequencing rationale

The five round-table experts (vision pipeline, frontend/PWA, storage, quality, delivery) agreed on far more than they disagreed. Where they conflicted, riskiest-first won, constrained by "every phase ends demoable or verifiable".

**Unanimous, adopted as-is:**

- **Deploy-to-HTTPS on day one.** Camera, WebGPU, OPFS, and wake lock all require a secure context, and the only hardware that matters is a real phone. The wrangler deploy loop is Phase 1, not a launch task.
- **The phone camera → WebGPU → readback path is the top technical risk.** `importExternalTexture` from a live stream, `requestVideoFrameCallback` timestamp semantics, and sustained 60 fps `mapAsync` readback are architecture-determining unknowns that cannot be mocked. They get a dedicated spike (Phase 2) on the Android target before any dependent code exists, with a written device-matrix report deciding the texture-import path and the timestamp source of truth. (The round table originally centered this risk on iOS Safari 26; ADR 0006 demoted iOS to best-effort, which shrinks the risk considerably — Chromium's WebGPU camera path is mature — but the spike stays, since the numbers still decide the architecture.)
- **Fixtures before tuning.** The fixture recorder ships in the very first camera-capable build so every hand-wave and flight harvests regression data; real tiny-whoop footage is the scarcest verification asset in the project and cannot be captured retroactively.
- **GPU reduces, CPU decides — exploited for sequencing.** The crossing state machine is pure TS, developed and tuned offline against recorded fixtures (Phase 4), not by repeated trips to the basement.

**Conflicts resolved (original round table):**

- **Pure-TS core first (quality) vs. GPU/device risk first (vision, delivery).** Device risk wins: pure TS carries near-zero platform risk, whereas a Safari WebGPU failure invalidates the architecture. Quality's key insight is preserved — the fixture formats and injectable-clock discipline are established in Phase 3 *before* the state machine exists, so the Phase 4 detector is spec'd as executable test cases.
- **Storage interface before anything writes (storage) vs. persistence late (delivery).** Both honored: nothing before Phase 6 persists user data (Phase 5's timer is deliberately amnesiac), and within Phase 6 the schema contract is frozen before the first OPFS byte. The lint-enforced storage seam and the real-browser Chromium + WebKit test rig land in Phase 1.
- **Service worker early (frontend) vs. PWA last (vision, delivery).** SW skeleton with explicit update-prompt flow lands in Phase 1 so months of deploys exercise the update path; offline hardening and install-flow polish stay in Phase 7.
- **Throwaway plain-TS pipeline lab (vision) vs. Svelte shell from the start (frontend).** Merged: Svelte scaffold from day one; the pipeline lab is a permanent hidden route (`/lab`, alongside `/diag`) rendering thin Svelte wrappers over the framework-free pipeline.
- **Calibration UX timing.** The lab gets touch-draggable ROI in Phase 3 (prototyping the risky interaction early); the product setup screen ports it in Phase 6 once tunables are validated.

**Revisions from the expert critique round (all blocking/important issues addressed; none rejected):**

- **Raw ROI clips + annotation sidecars are the canonical corpus; strip-energy JSON is a regenerable derivative** (quality, blocking; vision). Live-recorded energies embed the tunables and pause behavior active at record time; Phase 4's tuning loop changes both. Phase 3 ships a fixture-regeneration tool (clip → decode → GPU → JSON) so JSON caches are re-derived whenever tunables move; annotations attach to clips/frame indices, never to energy JSON.
- **Deterministic replay is a hard requirement, not "where possible"** (vision, quality). `VideoFileSource` is built on WebCodecs `VideoDecoder` for frame-exact, timestamp-exact stepping; clips are stored lossless/near-lossless; a quantified live-vs-replay divergence bound is decided before any corpus is recorded. All downstream acceptance numbers (±1 frame, ±2 frame E2E) rest on this.
- **One canonical WGSL path behind a thin per-source import pass** (vision). The Phase 2 texture decision may legitimately differ per platform (`texture_external` vs copied `texture_2d`); Phase 3 therefore normalizes every source into one canonical luminance texture so the golden-tested shader chain is the shader chain that runs live — avoiding exactly the double-implementation drift ADR 0002 exists to prevent.
- **The 30+ event fly-through corpus is decoupled from Phase 3's exit** (delivery, quality). It is a calendar-dependent logistics item, not an engineering task. Phase 3 exits on golden tests, replay fidelity (hand-wave fixtures), recorder, and the on-device self-test; corpus recording is a rolling parallel track that is an *entry criterion for Phase 4's acceptance and tuning loop* (items which need it), not for Phase 4's start (pure-TS spec/implementation work runs on synthetics). A short **race-speed pilot recording** inside Phase 3 acts as the ADR 0003 go/no-go gate *before* the full corpus session, so the scarce asset isn't captured with an inadequate setup.
- **Corpus acceptance is tiered** (delivery, quality; vision's escalation clause). Annotations carry a tier: `must-pass` (clean fly-throughs both directions, hand-waves, obvious noise) requiring 100% detection / zero false positives, vs `known-limitation` clips (simultaneous objects, extreme lighting, near-invisible whoop) where acceptance is documented behavior consistent with `detection.md`'s mitigation story. CI fails on must-pass regressions *and* on unexpected passes of known-hard cases (progress gets ratcheted in), so harvesting a hard field fixture never breaks the build — keeping the harvest loop alive. An explicit escalation path (capture guidance / reclassification with rationale / ADR 0003 revision) prevents an open-ended gate.
- **Phase 2 gets quantitative go/no-go thresholds and a named pivot ladder** (delivery). Minimum granted fps, maximum sustained readback latency, maximum timestamp jitter — declared before the spike starts, so "fail" has a defined next action.
- **Phase 2's probe surface is widened to every un-mockable behavior later phases depend on** (frontend, storage, vision): camera auto-exposure/focus/white-balance lockability (drives whether global-transient rejection is mandatory in Phase 4), speech `cancel()`/rapid-utterance/re-prime-after-background behavior (drives Phase 5's queue policy), `createWritable` abort/kill atomicity on real devices (drives Phase 6's atomic-write ADR), GPU `device.lost` behavior on background/foreground, and iOS tab-vs-installed-PWA OPFS partitioning.
- **The iOS partitioning decision is made at the end of Phase 2, not Phase 7** (frontend, storage; best-effort scope per ADR 0006, and skipped entirely if no iOS device is available). Home-screen web apps on iOS get storage separate from the Safari tab, and Safari applies eviction pressure to non-installed apps; deciding after Phase 6 data exists forces a data-rescue exercise. Phase 6 ships the resulting iOS guidance ("install before first data" banner) plus **working export** as the migration/backup path; Phase 7 keeps only install-flow polish.
- **Export moves from Phase 7 stub to Phase 6 working implementation** (storage). Real user data starts accumulating in Phase 6; export is the trivial read side and is the only backup for origin-scoped storage. The envelope-vs-file schemaVersion decision folds into Phase 6's schema-contract freeze. Import, merge, fuzz tests, and the nudge stay in Phase 7.
- **Domain data shapes freeze in Phase 4/5, not Phase 6** (storage). The session-semantics layer and in-memory session model use `storage.md`-shaped Course/Session/Lap types verbatim (defined in `src/core`, no persistence attached), so Phase 6 adds only envelopes, validators, and migrations around already-field-validated shapes.
- **The core→UI reactivity bridge is established in Phase 3, with an explicit per-frame rule** (frontend): per-frame data draws directly to canvas from the frame loop; Svelte `$state` is reserved for low-frequency state. Phase 5's running clock is rAF-driven. Phase 6 extends the proven pattern to storage repositories.
- **GPU device loss is handled end to end** (frontend): observed in Phase 2, recreate path built and tested in Phase 3, armed-session recovery in Phase 5.
- **Wake lock is acquired in `/lab` from Phase 3** (frontend) — the Phase 3/4 field sessions are minutes of hands-off capture; screen auto-lock would kill them. Phase 5 generalizes this into the product service.
- **EMA alpha is defined per unit time (dt-scaled), and replay honors recorded frame gaps** (vision) — otherwise tunables validated at 60 fps don't transfer to 30 fps devices and any dropped frame diverges live from replay. Strip energy is an integer hot-pixel count (`u32` atomicAdd) so GPU determinism is exact, not tolerance-based.
- **An on-device GPU self-test panel is the only possible automated GPU check on real iOS** (quality): `/diag` runs the embedded golden vectors and one bundled clip through the device's own GPU and reports max deviation. Phase 3 exit criterion on both phones; Phase 7 device-matrix line item.
- **A full-loop automated timer test lands in Phase 5, not Phase 7** (quality): annotated clip → pipeline → state machine → session layer → announcer interface, asserting laps, durations, discards, and exact announcement decisions. Phase 6 extends it with a deliberately slow/failing storage fake proving writes never delay timing or speech.
- **Phase 6 gets a mid-phase demoable checkpoint** (fly a persisted session, kill the tab, laps survive) **and sheds the desktop layout pass to Phase 7**, where desktop's real story (import-and-review) exists (delivery).

**Spec gaps assigned to phases** (each phase resolves its gaps and writes the answer back into the spec or an ADR, per CLAUDE.md):

- Phase 2: texture-import path per platform, capture-timestamp source of truth + fallback, camera auto-control lockability, speech queue feasibility, atomic-write device behavior, iOS OPFS partitioning posture (decision recorded here).
- Phase 3: canonical-texture normalization design (amend `detection.md`'s "frames enter WebGPU as external textures" if the copy path is required anywhere), strip-energy normalization, dt-scaled EMA, versioned fixture formats + clip codec + repo storage, GPU-in-CI confirmation, screen-orientation policy (app-state binding, not OS lock).
- Phase 4: precise wave-detector algorithm (hysteresis, plausibility window, gate-center strip for even N, direction frame of reference), global-transient rejection, trigger-level auto-suggestion, EMA-pause feedback contract with latency + max-pause timeout, test-mode debounce behavior.
- Phase 5: speech overlap/queue policy (from Phase 2 evidence), wake-lock scope, armed-session interruption behavior (including GPU device loss).
- Phase 6: atomic-write mechanism ADR (citing Phase 2 device evidence), app-level settings enumeration, multi-tab policy, session-file creation timing, export-envelope versioning.
- Phase 7: newer-version import refusal, orphan-session import policy, settings merge on import, "recent export" definition for the backup nudge.

## Exit criteria per phase (summary — details in each phase doc)

**Phase 1 — Foundation:** deploy loop + capability gate (adapter *and* device probe; `/diag`+`/lab` exempt) + seam lint (with `src/core/storage/` as the deliberate OPFS exemption surface) + SW skeleton with visible build id and one-tap update + CI green on node unit, Chromium (+ WebKit, informational) OPFS browser tests, and headless-Chromium WebGPU compute; the Android target phone procured (iOS device optional).

**Phase 2 — Device spike:** all `/diag` panels green on the Android device *against the pre-declared thresholds*; device-matrix report (ADR) records texture path, timestamp source, camera-control lockability, speech behavior, atomic-write behavior, device-loss behavior — plus iOS measurements and the iOS partitioning posture if a device is available.

**Phase 3 — Pipeline + lab + fixture tooling:** golden + determinism (bit-exact) node tests green in CI; deterministic replay (same clip → bit-identical FrameSample sequence) proven; `/lab` working on the Android phone (iOS best-effort) with wake lock and the CPU self-test passing; recorder (ring + continuous modes) proven including export off the device; field items tracked, not code-gated (ROI-cropped S22 cost re-measurement, race-speed pilot recording → ADR 0003 go/no-go, rolling corpus recording).

**Phase 4 — Crossing detection:** `detection.md` updated with the full detector spec; synthetic suite (incl. frame drops, jitter, global transients, delayed pause) green; must-pass corpus tier: 100% detection, correct directions, timestamps ±1 frame, zero false positives; known-limitation tier: documented behavior; test mode behaves correctly at the real gate. Corpus completeness is the entry criterion for the acceptance/tuning items only.

**Phase 5 — Flyable timer:** full-loop CI test green; real flying sessions with spoken laps matching stopwatch/video; 20-minute soak (thermal, memory, device-loss recovery) clean; storage.md-shaped domain types frozen.

**Phase 6 — Persistence + product UI:** contract suite green on MemoryStorage + OpfsStorage (Chromium; WebKit informational) incl. crash-simulation and quarantine; mid-phase checkpoint: tab kill mid-session loses at most the last lap; slow-storage fake proves the never-block contract; working export; iOS install-before-data guidance live; course CRUD, productized setup, session/course review all work on-device.

**Phase 7 — v1:** airplane-mode installed-PWA full session on Android (iOS best-effort); export/import round trip with fuzz-tested import; Playwright E2E green; desktop layout pass with the import-and-review story; device-matrix checklist (incl. on-device GPU self-test) and field acceptance protocol signed off; runbooks committed.
