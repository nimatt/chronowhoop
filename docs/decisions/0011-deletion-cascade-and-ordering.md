# 0011 — Deletion: cascade, ordering, and a bounded intent marker

**Status:** accepted, 2026-07-14. **Supplements [0010](0010-atomic-writes-opfs.md)** — the resume is a *new* startup step, not the startup sweep 0010's amendment retired; 0010 stands unedited (see decision 6). Builds on [0004](0004-opfs-storage.md) (one `Storage` seam; export/import as the cross-device story).

## Context

`product.md` said deletion was out of scope. A pilot who cannot delete a mistaken course or a session flown with the ROI in the wrong place accumulates junk they can never remove from a store they cannot browse — and every export carries it forever. Deletion has to exist, which forces four questions the storage layer cannot dodge: what happens to a deleted course's sessions, in what order the files go, what a crash mid-deletion leaves behind, and whether "deleted" is recorded anywhere.

The constraint that decides most of it is [0010](0010-atomic-writes-opfs.md): **atomicity is per-file, and there is no multi-file transaction.** A course deletion touches `courses.json` and N session files. It cannot be made atomic. The only thing left to choose is *which* crash state it can leave behind.

## Decision

1. **Cascade, not orphaning.** Deleting a course deletes every session whose `courseId` matches. Orphaning them would be a ghost bug shipped on purpose: no screen would ever list those sessions again (Home renders only known courses; the course view filters by `courseId`), but `exportAll` has **no such filter** — so they would ride out in every export, come back as "Unknown course" on every import, and count against quota forever. That is data the user believes is gone, that nothing displays, that nothing can remove, and that keeps returning. The existing orphan concept (`storage.md`) stays what it was meant to be: an *import-only* tolerance, plus the one file a cascade genuinely cannot condemn (an unreadable session, whose `courseId` lives in bytes we could not parse).

2. **Hard, immediate, permanent.** No trash, no soft-delete, no undo. The recovery path is the **export file** — the same file that already carries data from phone to desktop — and the confirmation screen offers to take one first, warning honestly when what is about to go was never exported (or was flown since the last export). A trash would be a second store to browse, sweep, quota-manage and back up, for a product whose whole persistence story is "one JSON file you own".

3. **Sessions before `courses.json`.** A direct consequence of 0010: the order is `[INTENT]` write `courses.json` with the course still present and a pending-deletion marker added → remove the session files → `[COMMIT]` re-read `courses.json` and write it back without the course and without the marker. Every read that can reject (`loadCourses` can throw `unsupported-version`) runs **before** the first write. The two possible crash states are: *course still present, some of its sessions gone* (this order) versus *sessions still on disk, their course gone* (the reverse). The first is visible-and-wrong: the app shows a course, its session list is short, and the marker says so. The second is invisible-and-retained: files nothing lists, nothing can remove, and every export still ships. Visible-and-wrong is recoverable; invisible-and-retained is decision 1's ghost bug arriving through the back door. The commit write is a single write that drops the course, its marker, and `lastCourseId` together, so no crash can land between "course gone" and "marker gone".

4. **The marker is a bounded work list, not an instruction.** `settings.pendingCourseDeletions[]` records `{ courseId, courseName, sessionIds }` — the **exact session ids the user saw counted and confirmed**. A resume deletes those ids and nothing else. Re-deriving the doomed set at resume time (filter live sessions by `courseId`) would look equivalent and is not: it turns an abandoned deletion into an *unbounded standing instruction*. Fail a delete, walk away, fly the course for another month, and the next launch sweeps up sessions that did not exist when the confirmation was given. Hence also the **`flown-since` abandonment rule**: if the resume finds a live session on the course that the marker does not name, the deletion is abandoned, the marker cleared, and the course kept intact — we never destroy data the confirmation did not count. Deleting again is two taps and re-states the real, current blast radius.

5. **No PERSISTED tombstones.** Nothing on disk records that anything was deleted, so re-importing an older export **re-adds** the deleted items. That is the restore path, not a bug — it is the *only* undo this product has (decision 2), and it is exactly what decision 2 tells the user to rely on. Tombstones would make the import silently drop sessions the user just handed it, on the very phone→desktop path 0004 calls the v1 cross-device story. **The deliberate exception is the in-memory, instance-scoped resurrection guard**: a `Storage` instance remembers the session and course ids it deleted and rejects a `saveSession` for them with `not-found` for its own lifetime. It exists because the live-session persister writes fire-and-forget, outlives its component, and `saveSession` creates the file — so a straggling tail write could otherwise re-create a session file whose course is already gone. It is *in-memory and instance-scoped, in those words*: persisting it to close the apparent inconsistency would destroy the only undo the product has, and `importAll` re-admits the ids in its envelope for the same reason.
   **The marker never leaves the store either.** The export strips `pendingCourseDeletions` from the envelope's settings — the marker says "a destruction of this course is in flight *here*", which is state of this store, not user data, and no export file should advertise it or carry that instruction into the store it lands in. And an import **abandons** every marker naming a course, or a session, the file carries: a marker that survived a restore would be replayed by the next launch against *exactly* the ids the file just put back — no strays, so the `flown-since` rule cannot fire, and the resume would finish the cascade over the restored data. The only undo this product has would undo itself one launch later, in silence. Abandoning costs a re-confirmation of a deletion that can still be performed; completing it costs the data the user went and got back.

6. **The resume supplements 0010's decision 3 — it does not revive it.** 0010 specified a startup sweep of stray files, and its 2026-07-13 amendment retired that sweep because *"our writes create no artifacts of their own"*. **That remains true, and it is not what this phase changes.** "Artifacts" there means *stray files* — the amendment enumerates them: browser-managed `.crswap` staging files, deliberately-kept quarantine copies, non-`.json` names reads already skip — and a cascade creates none of them. The intent marker is not a stray file: it is a value inside a file we deliberately own and write. What *is* newly true is narrower — for the first time, something runs at startup. It is not that sweep. The sweep would have inferred intent from filenames nobody wrote on purpose; the resume replays an intent we recorded on purpose, in `courses.json`, bounded to a work list the user confirmed. It never guesses, and it can only ever remove ids that confirmation named. **0010's decision 3 stays retired, 0010 is not edited, and `opfs-storage.ts`'s "there is deliberately NO startup sweep" is correct as written** — the resume is not one.

## Consequences

- **The `Storage` seam grows three members** (`deleteSession`, `deleteCourse`, `resumePendingDeletions`), which breaks every hand-written `Storage` double at compile time. That is the point: they enumerate everything claiming to implement the seam.
- **A partially-applied cascade is a first-class state**, not an error: the marker makes it self-describing, the resume makes it self-healing, and the user is told which of the two outcomes (`completed` / `abandoned`) landed.
- **`pendingCourseDeletions` needs no `SCHEMA_VERSION` bump** — an added optional key is forward-compatible on read, and an older app drops it on its next write, degrading to the un-marked partial state (a course short of sessions), which fails safe.
- **`courses.json` now has four read-merge-write callers** (course saves, settings, delete, resume) plus import and export. All six share one serialization queue: two unsynchronized read-merge-write cycles do not merely race — the loser's document is overwritten from a stale snapshot, which resurrects a deleted course whose sessions are genuinely destroyed.
- **An unreadable session file can outlive its course.** It cannot be attributed to one, so a cascade cannot condemn it; it becomes an orphan, and if it is `unsupported-version` it keeps failing a strict export even after its course is gone. Accepted and documented (`storage.md`), not fixed — there is no honest way to destroy a file on the strength of bytes we could not parse.
- **The export became load-bearing**, so it was hardened in the same change: a session file that vanishes mid-scan now fails a strict export instead of being silently omitted, export filenames carry seconds, and the envelope's settings no longer carry an in-flight deletion marker — "Export backup first" must not be able to hand out a truncated backup, overwrite the previous one within the same minute, or ship a file that tells the next store to finish destroying what it contains.
- **Cross-device delete propagation does not exist.** Deleting on the phone does not delete on the desktop; that is sync, an explicit non-goal.
